---

## 1. Overview

AskHiDock adds natural language query capability to HiDockSkill. Users can ask questions about their meetings and receive accurate, sourced answers drawn from the entire meeting history across all tiers (hot/warm/cold).

**Core Principle:** No new infrastructure. AskHiDock uses the same Ollama instance already running for summarization, the same Markdown files already generated by the sync pipeline, and adds only SQLite as a local search index. Everything stays 100% local.

```
User: "What did John ask me to deliver this week?"

AskHiDock: "John asked you to deliver the Q3 report with updated cloud 
cost projections by Friday April 4. This was assigned during the Q3 
Planning meeting on April 3rd."

[source: meetings/hotmem/202604/20260403-q3-planning.md @10:34]
```

---

## 2. Problem Statement

HiDockSkill today captures, transcribes, summarizes, and visualizes meetings. But users cannot query their meeting history. To find "what John said about the budget," they must manually browse the Galaxy Dashboard or grep Markdown files.

Competitors (Plaud, Otter) offer "Ask AI" features but use stateless RAG — each query starts from scratch with no accumulated knowledge. HiDockSkill's existing tiered wiki structure (hot/warm/cold with structured Markdown) is a superior foundation for retrieval, but it lacks a query interface.

**The gap:** HiDockSkill has the knowledge. It lacks the retrieval.

---

## 3. Design Constraints

1. **100% local.** No cloud APIs. No external embedding services. Everything runs through Ollama.
2. **No new infrastructure.** No separate search servers, no graph databases, no QMD sidecar. SQLite only.
3. **Additive to existing codebase.** No changes to existing sync pipeline, storage layout, or Galaxy Dashboard. New code goes in `src/search/`.
4. **Same Ollama model.** Uses the already-configured model (default: `qwen3.5:9b`) for query analysis and synthesis. Uses Ollama embedding endpoint for vectors.
5. **Performance target.** < 5 seconds end-to-end for typical queries on Apple Silicon hardware.

## 5. Detailed Design

### 5.1 Search Index (SQLite)

A single SQLite database (`<storage>/search.db`) stores all search data. Created and updated during `meetings:sync`.

**Schema:**

```sql
-- Chunks: searchable segments of meeting notes
CREATE TABLE chunks (
  id TEXT PRIMARY KEY,
  file_path TEXT NOT NULL,           -- relative to storage root
  tier TEXT NOT NULL,                -- 'hot', 'warm', 'cold'
  meeting_date TEXT NOT NULL,        -- ISO date
  source_type TEXT,                  -- 'room', 'call', 'whisper'
  chunk_type TEXT NOT NULL,          -- 'summary', 'action_item', 'decision',
                                     -- 'topic', 'transcript', 'key_point'
  content TEXT NOT NULL,             -- chunk text
  speakers TEXT,                     -- JSON array of speaker names
  embedding BLOB                    -- Float32Array as blob
);

-- FTS5 virtual table for BM25 keyword search
CREATE VIRTUAL TABLE chunks_fts USING fts5(
  content, 
  speakers,
  content=chunks, 
  content_rowid=rowid,
  tokenize='porter unicode61'
);

-- Entities: people, projects, topics
CREATE TABLE entities (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  type TEXT NOT NULL,                -- 'person', 'project', 'topic'
  first_seen TEXT,
  last_seen TEXT,
  meeting_count INTEGER DEFAULT 0,
  metadata TEXT                      -- JSON blob for extra attributes
);

-- Entity-meeting junction
CREATE TABLE entity_meetings (
  entity_id TEXT REFERENCES entities(id),
  file_path TEXT,
  meeting_date TEXT,
  role TEXT,                         -- 'attendee', 'mentioned', 'assignee'
  PRIMARY KEY (entity_id, file_path)
);

-- Action items (extracted from front matter)
CREATE TABLE action_items (
  id TEXT PRIMARY KEY,
  text TEXT NOT NULL,
  assignee TEXT,
  assigner TEXT,
  due_date TEXT,
  status TEXT DEFAULT 'open',        -- 'open', 'done', 'overdue'
  source_file TEXT NOT NULL,
  source_date TEXT NOT NULL,
  entity_id TEXT REFERENCES entities(id)
);

-- Decisions (extracted from front matter)  
CREATE TABLE decisions (
  id TEXT PRIMARY KEY,
  text TEXT NOT NULL,
  decided_by TEXT,
  source_file TEXT NOT NULL,
  source_date TEXT NOT NULL
);
```

### 5.2 Indexing Pipeline

Runs at the end of every `meetings:sync` cycle. No separate indexing step.

```typescript
// Pseudocode for indexer.ts

async function indexNote(notePath: string): Promise<void> {
  // 1. Parse Markdown + YAML front matter
  const note = parseMarkdownNote(notePath);
  
  // 2. Extract chunks by type
  const chunks: SearchChunk[] = [
    { type: 'summary',     content: note.summary },
    { type: 'key_point',   content: note.keyPoints.join('\n') },
    ...note.actionItems.map(a => ({ type: 'action_item', content: a.text })),
    ...note.decisions.map(d => ({ type: 'decision', content: d.text })),
    ...note.topics.map(t => ({ type: 'topic', content: t })),
    // Transcript chunked into ~500-token segments
    ...chunkTranscript(note.transcript, 500),
  ];
  
  // 3. Generate embeddings for each chunk (batched Ollama call)
  const embeddings = await ollamaEmbed(chunks.map(c => c.content));
  
  // 4. Insert into SQLite (chunks + FTS5 + entities)
  db.transaction(() => {
    for (const [i, chunk] of chunks.entries()) {
      insertChunk({ ...chunk, embedding: embeddings[i] });
    }
    updateEntities(note.metadata);
    upsertActionItems(note.actionItems, notePath);
    upsertDecisions(note.decisions, notePath);
  })();
}
```

**Embedding model:** `nomic-embed-text` via Ollama (274MB download, 768-dim vectors).

> **[v2 change — Staff Review Issue 6]** The embedding model is NOT auto-pulled. On first sync, the system checks for the model and warns the user. Search gracefully degrades to BM25-only mode if embeddings are unavailable.

### 5.3 Query Analyzer (Two-Layer: Deterministic + LLM)

> **[v2 change — Staff Review Issue 1]** The query analyzer uses a two-layer approach. A deterministic pre-parser handles common patterns reliably (time expressions, known entity names, action item queries). The LLM analyzer handles ambiguous queries. If the LLM produces invalid JSON, the system falls back to the deterministic parse + raw BM25/vector search. The query never fails silently.

```typescript
// query-analyzer.ts

interface SearchPlan {
  keywords: string[];              // BM25 search terms
  semantic_query: string;          // Vector search query
  entity_filters: {
    speakers?: string[];           // filter by speaker name
    projects?: string[];           // filter by project name
  };
  time_filter?: {
    start?: string;                // ISO date
    end?: string;                  // ISO date
    relative?: string;             // 'today', 'this_week', 'this_month', 'this_quarter'
  };
  chunk_types?: string[];          // 'action_item', 'decision', 'summary', etc.
  strategy: 'entity_first' | 'search_first' | 'browse' | 'timeline';
  max_results: number;
}
```

**Layer 1: Deterministic Pre-Parser (always runs, <10ms):**

```typescript
function preParseQuery(query: string, entityStore: EntityStore): Partial<SearchPlan> {
  const plan: Partial<SearchPlan> = {};
  
  // Time expressions (regex, no LLM)
  if (/yesterday/i.test(query)) plan.time_filter = { relative: 'yesterday' };
  else if (/today/i.test(query)) plan.time_filter = { relative: 'today' };
  else if (/this week/i.test(query)) plan.time_filter = { relative: 'this_week' };
  else if (/last week/i.test(query)) plan.time_filter = { relative: 'last_week' };
  else if (/this month/i.test(query)) plan.time_filter = { relative: 'this_month' };
  else if (/this quarter/i.test(query)) plan.time_filter = { relative: 'this_quarter' };
  
  // Match against known entity names in the entity store
  const knownPeople = entityStore.getAllPeople();
  for (const person of knownPeople) {
    if (query.toLowerCase().includes(person.name.toLowerCase())) {
      plan.entity_filters = { speakers: [person.name] };
      plan.strategy = 'entity_first';
    }
  }
  
  // Chunk type detection (keyword patterns)
  if (/action item|todo|deliver|owe|committed|due|overdue|assigned/i.test(query)) {
    plan.chunk_types = ['action_item'];
  } else if (/decide|decision|chose|approved|rejected|agreed/i.test(query)) {
    plan.chunk_types = ['decision'];
  } else if (/summarize|summary|overview|recap|what happened/i.test(query)) {
    plan.strategy = 'browse';
  } else if (/history|evolved|changed|first time|when did/i.test(query)) {
    plan.strategy = 'timeline';
  }
  
  return plan;
}
```

**Layer 2: LLM Analyzer (runs for ambiguous queries, ~1s):**

```typescript
const ANALYZER_PROMPT = `You are a query analyzer for a meeting knowledge base.
Given a user's natural language question, output a JSON search plan.

Available chunk types: summary, action_item, decision, topic, key_point, transcript
Available strategies:
  - entity_first: when query is about a specific person/project
  - search_first: when query is about a topic or keyword
  - browse: when query is broad ("summarize this week")
  - timeline: when query asks about evolution/history

Respond ONLY with valid JSON. No explanation.

User query: {query}
Today's date: {today}`;
```

**Merge logic:**

```typescript
async function analyzeQuery(query: string): Promise<SearchPlan> {
  // Layer 1: deterministic (always works)
  const deterministicPlan = preParseQuery(query, entityStore);
  
  // Layer 2: LLM (may fail)
  let llmPlan: Partial<SearchPlan> = {};
  try {
    const response = await ollama.chat({ /* analyzer prompt */ });
    llmPlan = JSON.parse(response.message.content);
  } catch (e) {
    // LLM failed or returned invalid JSON — proceed with deterministic only
    console.warn('LLM analyzer failed, using deterministic parse:', e.message);
  }
  
  // Merge: deterministic wins on conflicts (it's more reliable)
  return {
    keywords: llmPlan.keywords || query.split(/\s+/),
    semantic_query: llmPlan.semantic_query || query,
    entity_filters: deterministicPlan.entity_filters || llmPlan.entity_filters || {},
    time_filter: deterministicPlan.time_filter || llmPlan.time_filter,
    chunk_types: deterministicPlan.chunk_types || llmPlan.chunk_types,
    strategy: deterministicPlan.strategy || llmPlan.strategy || 'search_first',
    max_results: 20,  // retrieve 20, pack to token budget later
  };
}
```

### 5.4 Search Engine

Parallel execution of three search paths, merged via Reciprocal Rank Fusion.

> **[v2 change — Staff Review Issue 3]** Vector search uses tiered strategy: search hot tier first (in-memory, fast). Only search warm/cold if hot results are insufficient. This prevents cold-start memory pressure at scale.

```typescript
// search-engine.ts

const CONFIDENCE_THRESHOLD = 0.4; // minimum RRF score to consider "sufficient"

async function search(plan: SearchPlan): Promise<RankedChunk[]> {
  // Run BM25 + entity lookup in parallel (these are always fast)
  const [bm25Results, entityResults] = await Promise.all([
    // Path 1: BM25 full-text search (all tiers, FTS5 is fast)
    ftsSearch(plan.keywords.join(' '), {
      speakers: plan.entity_filters?.speakers,
      dateRange: plan.time_filter,
      chunkTypes: plan.chunk_types,
      limit: 20,
    }),
    
    // Path 3: Entity store lookup (instant)
    entityLookup(plan.entity_filters, plan.time_filter),
  ]);
  
  // Path 2: Vector search — tiered (hot first, deep on demand)
  let vectorResults: RankedChunk[] = [];
  if (embeddingsAvailable) {
    // Always search hot tier (last 30 days, in-memory, <50ms)
    vectorResults = await vectorSearch(plan.semantic_query, {
      tier: 'hot',
      speakers: plan.entity_filters?.speakers,
      dateRange: plan.time_filter,
      chunkTypes: plan.chunk_types,
      limit: 20,
    });
    
    // Search deeper only if hot results are insufficient
    const hotIsEnough = vectorResults.length >= 5 
      && vectorResults[0]?.score >= CONFIDENCE_THRESHOLD;
      
    if (!hotIsEnough || plan.strategy === 'timeline') {
      const deepResults = await vectorSearch(plan.semantic_query, {
        tier: ['warm', 'cold'],
        speakers: plan.entity_filters?.speakers,
        dateRange: plan.time_filter,
        chunkTypes: plan.chunk_types,
        limit: 20,
      });
      vectorResults = [...vectorResults, ...deepResults];
    }
  }
  
  // Merge via Reciprocal Rank Fusion
  const merged = reciprocalRankFusion([bm25Results, vectorResults], 60);
  
  // Inject entity results (action items, decisions) as first-class results
  const combined = interleaveEntityResults(merged, entityResults);
  
  // Apply temporal decay
  const decayed = applyTemporalDecay(combined, halfLifeDays: 30);
  
  // Return top N
  return decayed.slice(0, plan.max_results);
}
```

**Reciprocal Rank Fusion (RRF):**

```typescript
// rrf.ts
function reciprocalRankFusion(
  resultSets: RankedChunk[][], 
  k: number = 60
): RankedChunk[] {
  const scores = new Map<string, number>();
  
  for (const results of resultSets) {
    for (let rank = 0; rank < results.length; rank++) {
      const id = results[rank].id;
      const current = scores.get(id) || 0;
      scores.set(id, current + 1 / (k + rank + 1));
    }
  }
  
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .map(([id, score]) => ({ ...getChunk(id), rrf_score: score }));
}
```

**Temporal Decay:**

```typescript
// Newer content ranks higher, but old critical decisions still surface
function applyTemporalDecay(
  chunks: RankedChunk[], 
  halfLifeDays: number = 30
): RankedChunk[] {
  const now = Date.now();
  return chunks.map(chunk => {
    const daysAgo = (now - new Date(chunk.meeting_date).getTime()) / 86400000;
    const decay = Math.pow(0.5, daysAgo / halfLifeDays);
    return { ...chunk, score: chunk.score * (0.3 + 0.7 * decay) };
    // Floor of 0.3 ensures old content is never fully suppressed
  }).sort((a, b) => b.score - a.score);
}
```

### 5.5 Synthesizer

Takes ranked chunks and produces a natural language answer. One Ollama call.

```typescript
// synthesizer.ts

const SYNTHESIZER_PROMPT = `You are an AI assistant answering questions about 
the user's meeting history. You have been given relevant chunks from their 
meeting notes.

Rules:
1. Answer based ONLY on the provided chunks. Do not hallucinate.
2. Cite sources using [source: filename @timestamp] format.
3. If the chunks don't contain enough information, say so explicitly.
4. Be concise. Lead with the direct answer, then provide context.
5. If multiple meetings are relevant, present them chronologically.

User question: {query}

Relevant meeting data:
{chunks}

Answer:`;

async function synthesize(
  query: string, 
  chunks: RankedChunk[]
): Promise<Answer> {
  // [v2 change — Staff Review Issue 4] Hard token budget prevents context overflow
  const MAX_SYNTHESIS_TOKENS = 3000;
  let tokenCount = 0;
  const selected: RankedChunk[] = [];
  
  for (const chunk of chunks) {
    const chunkTokens = Math.ceil(chunk.content.length / 4); // rough estimate
    if (tokenCount + chunkTokens > MAX_SYNTHESIS_TOKENS) break;
    selected.push(chunk);
    tokenCount += chunkTokens;
  }
  
  const context = selected.map(c => 
    `[${c.meeting_date} | ${c.file_path} | ${c.chunk_type}]\n${c.content}`
  ).join('\n---\n');
  
  const response = await ollama.chat({
    model: process.env.OLLAMA_MODEL || 'qwen3.5:9b',
    messages: [{ 
      role: 'user', 
      content: SYNTHESIZER_PROMPT
        .replace('{query}', query)
        .replace('{chunks}', context)
    }],
  });
  
  return {
    text: response.message.content,
    sources: chunks.map(c => ({
      file: c.file_path,
      date: c.meeting_date,
      type: c.chunk_type,
    })),
  };
}
```

### 5.6 Entity Store (Two-Phase: Deterministic + LLM Enrichment)

> **[v2 change — Staff Review Issue 2]** Entity extraction now has two phases. Phase 1 is deterministic (from YAML front matter — same as v1). Phase 2 adds one Ollama call during sync to extract relationships, role changes, and resolved commitments. Phase 2 runs async and does not block the sync pipeline.

**Phase 1: Deterministic (from YAML front matter, no LLM):**

```typescript
// entities.ts

function updateEntities(metadata: MeetingMetadata): void {
  // Upsert each attendee
  for (const speaker of metadata.speakers) {
    db.prepare(`
      INSERT INTO entities (id, name, type, first_seen, last_seen, meeting_count)
      VALUES (?, ?, 'person', ?, ?, 1)
      ON CONFLICT(id) DO UPDATE SET
        last_seen = MAX(last_seen, excluded.last_seen),
        meeting_count = meeting_count + 1
    `).run(
      slugify(speaker), speaker, metadata.date, metadata.date
    );
    
    // Link entity to meeting
    db.prepare(`
      INSERT OR IGNORE INTO entity_meetings (entity_id, file_path, meeting_date, role)
      VALUES (?, ?, ?, 'attendee')
    `).run(slugify(speaker), metadata.filePath, metadata.date);
  }
  
  // Upsert topics as entities
  for (const topic of metadata.topics) {
    db.prepare(`
      INSERT INTO entities (id, name, type, first_seen, last_seen, meeting_count)
      VALUES (?, ?, 'topic', ?, ?, 1)
      ON CONFLICT(id) DO UPDATE SET
        last_seen = MAX(last_seen, excluded.last_seen),
        meeting_count = meeting_count + 1
    `).run(slugify(topic), topic, metadata.date, metadata.date);
  }
}
```

**Phase 2: LLM Enrichment (one Ollama call per meeting, async):**

```typescript
// entity-enrichment.ts

const ENRICHMENT_PROMPT = `Extract structured entity information from this meeting 
summary. Output ONLY valid JSON.

{
  "people": [
    { "name": "full name", "role": "title if mentioned", "aliases": ["short names"] }
  ],
  "relationships": [
    { "from": "person", "relation": "reports_to|manages|collaborates_with|assigned_to", "to": "person or project" }
  ],
  "commitments": [
    { "who": "person", "what": "task description", "due": "absolute YYYY-MM-DD or null", "assigned_by": "person or null" }
  ],
  "project_updates": [
    { "project": "name", "status": "on_track|at_risk|blocked|completed", "owner": "person or null" }
  ]
}

Meeting date: {date}
Meeting summary:
{summary}`;

async function enrichEntities(notePath: string, summary: string, date: string): Promise<void> {
  try {
    const response = await ollama.chat({
      model: process.env.OLLAMA_MODEL || 'qwen3.5:9b',
      messages: [{ role: 'user', content: ENRICHMENT_PROMPT
        .replace('{date}', date)
        .replace('{summary}', summary) 
      }],
    });
    
    const enrichment = JSON.parse(response.message.content);
    
    // Update entity roles
    for (const person of enrichment.people || []) {
      if (person.role) {
        db.prepare(`UPDATE entities SET metadata = json_set(
          COALESCE(metadata, '{}'), '$.role', ?, '$.role_as_of', ?
        ) WHERE name = ?`).run(person.role, date, person.name);
      }
    }
    
    // Store relationships
    for (const rel of enrichment.relationships || []) {
      db.prepare(`INSERT OR REPLACE INTO entity_relationships 
        (from_entity, relation, to_entity, observed_date, source_file)
        VALUES (?, ?, ?, ?, ?)`
      ).run(slugify(rel.from), rel.relation, slugify(rel.to), date, notePath);
    }
    
    // Store commitments with resolved dates
    for (const commit of enrichment.commitments || []) {
      db.prepare(`INSERT OR IGNORE INTO action_items
        (id, text, assignee, assigner, due_date, status, source_file, source_date)
        VALUES (?, ?, ?, ?, ?, 'open', ?, ?)`
      ).run(
        `${slugify(commit.who)}-${date}-${slugify(commit.what).slice(0,20)}`,
        commit.what, commit.who, commit.assigned_by, 
        commit.due, notePath, date
      );
    }
  } catch (e) {
    // Enrichment failure is non-fatal — deterministic data is already stored
    console.warn('Entity enrichment failed (non-blocking):', e.message);
  }
}
```

| Query | Entity Store Behavior |
|---|---|
| "What has Sarah committed to?" | `SELECT * FROM action_items WHERE assignee LIKE '%Sarah%' AND status = 'open'` |
| "When did we first discuss pricing?" | `SELECT first_seen FROM entities WHERE name LIKE '%pricing%'` |
| "Who have I met with most this month?" | `SELECT name, meeting_count FROM entities WHERE type = 'person' ORDER BY meeting_count DESC` |
| "Open action items overdue" | `SELECT * FROM action_items WHERE status = 'overdue'` |

These return structured results instantly (no Ollama call needed for the lookup). The synthesizer uses them alongside search results to compose the final answer.

## 7. Query Types and Strategies

| Query Type | Example | Strategy | Primary Search Path |
|---|---|---|---|
| **Simple recall** | "What did John say about the deadline?" | entity_first | Entity lookup → BM25 → synthesize |
| **Time-filtered** | "Summarize yesterday's meetings" | browse | Date filter → read all matching notes → synthesize |
| **Action tracking** | "What are my open action items?" | entity_first | SQL query on action_items table → synthesize |
| **Cross-meeting** | "How has the pricing strategy evolved?" | timeline | Entity lookup for "pricing" → all meetings → chronological synthesis |
| **Fuzzy recall** | "That thing someone said about React" | search_first | Vector search (semantic) → BM25 backup → synthesize |
| **Person profile** | "Tell me about Sarah" | entity_first | Entity lookup → all meetings with Sarah → synthesize |
| **Decision audit** | "When did we decide to use Stripe?" | search_first | BM25 "Stripe" + decision chunk type → synthesize |
| **Contradiction** | "Did anyone change their mind about the launch date?" | timeline | BM25 "launch date" → chronological → synthesize with conflict detection |

## 8. Performance Characteristics

### 8.1 Latency Budget

| Step | Target | Method |
|---|---|---|
| Query analysis | <1s | Single Ollama call, small prompt |
| Embedding query | <200ms | Single Ollama embed call |
| BM25 search | <50ms | SQLite FTS5, indexed |
| Vector search | <200ms | Brute-force cosine sim in SQLite (sufficient for <100K chunks) |
| Entity lookup | <10ms | SQLite indexed query |
| RRF merge | <5ms | In-memory sort |
| Synthesis | <3s | Single Ollama call with ~2K token context |
| **Total** | **<5s** | |

### 8.2 Storage Overhead

| Component | Size per meeting | Size at 500 meetings | Size at 5000 meetings |
|---|---|---|---|
| FTS5 index | ~2KB | ~1MB | ~10MB |
| Embeddings (768-dim float32) | ~30KB (10 chunks) | ~15MB | ~150MB |
| Entity tables | ~500B | ~250KB | ~2.5MB |
| **Total** | ~33KB | ~16MB | ~163MB |

Negligible relative to the audio files (~43MB per meeting at MP3 96kbps).

### 8.3 Scaling Limits

At 5000 meetings (~2 years daily use), brute-force vector similarity across ~50K chunks takes ~200ms on Apple Silicon. This is acceptable.

At 25,000 meetings (~10 years), brute-force vector search will exceed 1s. At that scale, add HNSW approximate nearest neighbor index. This is a future optimization, not a launch blocker.

## 10. Error Handling

| Failure | Behavior |
|---|---|
| Ollama not running | Clear error message: "Start Ollama with `ollama serve`" |
| Embedding model not available | Warn user, continue with BM25-only mode (v2 change) |
| SQLite corruption | `search:rebuild` command regenerates from Markdown files |
| Empty search results | Synthesizer reports "No relevant meetings found" with suggestions |
| Ollama timeout | Retry once, then fall back to BM25-only results without synthesis |
| Query too vague | Query analyzer requests clarification instead of guessing |
| LLM returns invalid JSON | Deterministic pre-parser handles query; never fails silently (v2 change) |

## 12. Open Questions

1. **Embedding model choice:** `nomic-embed-text` (768-dim, 274MB) vs `all-minilm` (384-dim, 46MB). Nomic is higher quality but larger. Should we default to all-minilm for faster first-run experience?

2. **Chunk granularity:** Should transcripts be chunked by speaker turn, by time window (60s segments), or by topic boundary? Speaker turns preserve attribution but vary wildly in length.

3. **Index persistence across tier demotion:** When the monthly cycle condenses warm → cold pages, should we re-embed the condensed content, or keep the original chunk embeddings?

4. **Galaxy Dashboard search UX:** Inline answer panel vs modal vs separate page? Should matching nodes pulse/highlight in the constellation?

## Appendix A: Example Query Flow

**User asks:** "What has Sarah committed to this quarter?"

**Step 1 — Query Analyzer output:**
```json
{
  "keywords": ["Sarah", "committed", "action", "deliverable"],
  "semantic_query": "Sarah's commitments and action items this quarter",
  "entity_filters": { "speakers": ["Sarah"] },
  "time_filter": { "relative": "this_quarter" },
  "chunk_types": ["action_item", "decision", "summary"],
  "strategy": "entity_first",
  "max_results": 15
}
```

**Step 2 — Search results:**

Entity store returns:
```
action_items WHERE assignee LIKE '%Sarah%' AND source_date >= '2026-01-01':
  - "Ship Nova beta by April 15" (open, from 20260320-product-sync.md)
  - "Draft announcement for board" (open, due Apr 5, from 20260404-board.md)
  - "Hire 3 engineers by March 31" (done, from 20260115-planning.md)
  - "Complete Q1 engineering review" (done, from 20260301-standup.md)
```

BM25 returns 8 matching chunks mentioning Sarah + commitments.
Vector returns 12 semantically similar chunks.
RRF merge produces top 15 ranked chunks.

**Step 3 — Synthesized answer:**

```
Sarah has 2 open commitments this quarter:

1. Ship Nova beta by April 15 — currently on track, 78% complete
   [source: hotmem/202603/20260320-product-sync.md]

2. Draft board announcement by April 5 (today)
   [source: hotmem/202604/20260404-board-review.md]

She completed 2 earlier commitments:
- Hired 3 engineers by March 31 ✅

Sarah has attended 23 meetings this month across 4 active projects.
```

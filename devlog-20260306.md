# Dev Log - 2026-03-06

## Summary
Fixed the USB auto-detect behavior gap where P1 plug-in produced no prompt in normal usage by adding a runnable continuous watch CLI and tightening monitor startup semantics.

## Why this change
- Root cause: monitoring logic existed in library code but no standard runtime command started it, so OpenClaw flow could miss plug-in notifications.
- Requirement: notify on each `disconnected -> connected` transition and support optional startup emission when already connected.

## What changed
- Monitor behavior (`src/nodeUsb.ts`):
  - Added `emitOnStartupIfConnected?: boolean` to `HiDockConnectionMonitorOptions`.
  - Default is `true`.
  - Emission logic now:
    - first successful poll emits when startup option is `true`
    - first successful poll does not emit when option is `false`
    - later reconnect transitions always emit once
    - continuously connected state does not emit duplicates
- New runnable watcher CLI (`src/cli/usbWatch.ts`):
  - Added long-running USB watch command suitable for OpenClaw.
  - Added args:
    - `--interval-ms <n>`
    - `--emit-on-startup`
    - `--no-emit-on-startup`
  - Added clean shutdown on `SIGINT`/`SIGTERM`.
- Script wiring (`package.json`):
  - Added `usb:watch`: `npm run build && node dist/cli/usbWatch.js`
- Documentation (`README.md`):
  - Added explicit OpenClaw continuous watch usage and expected output.
- Tests:
  - Updated `tests/nodeUsb.test.ts` for startup emit default `true` and `false` behavior.
  - Added `tests/usbWatchCli.test.ts` for CLI argument wiring defaults/flags.

## Update - Thread-Aware Slack Routing (Activity Window)

### Why this change
- Follow-up requirement: on plug-in event, route to Slack thread only when the user is actively chatting there in the recent window; otherwise route to new main timeline message.

### What changed
- Routing config in `src/cli/usbWatch.ts`:
  - Added CLI/env settings:
    - `--slack-thread-id` / `HIDOCK_USB_WATCH_SLACK_THREAD_ID`
    - `--active-window-minutes` / `HIDOCK_USB_WATCH_ACTIVE_WINDOW_MINUTES` (default 5)
    - `--slack-activity-target` / `HIDOCK_USB_WATCH_SLACK_ACTIVITY_TARGET`
    - `--slack-activity-user-id` / `HIDOCK_USB_WATCH_SLACK_ACTIVITY_USER_ID`
- Implemented robust activity probe via OpenClaw CLI read:
  - `openclaw message read --channel slack --target <activityTarget> --limit 50 --json`
  - tolerant JSON parsing for common timestamp/thread/user field variants
- Implemented routing behavior:
  - active in-thread message within window -> send with `--reply-to <threadId>`
  - stale/inactive/read-failure -> send without `--reply-to`
- Preserved failure handling:
  - activity-read errors are non-fatal and fallback to non-thread send
  - send failures still bubble to existing `createUsbWatchPlugInHandler` catch+log path
- Tests (`tests/usbWatchCli.test.ts`):
  - expanded CLI parsing coverage for new flags/envs
  - added thread-routing decision tests (active, stale, read-error)
- Docs (`README.md`):
  - added thread-aware usage examples and updated startup log example

# Devlog 2026-04-08 — Speaker Rename: from heuristics to explicit choice

## TL;DR

Shipped the speaker rename feature in five iterations, each one fixing a problem
with the previous design's assumptions. Final design uses a modal popup for
ambiguous cases instead of trying to infer user intent from heuristics.

## What was built

A speaker rename feature that walks all four storage locations of speaker
identity in one operation:

1. **Note `## Transcript`** — line-anchored regex on `[<from>(\s+@<sec>)?]:`
2. **Note `## Summary`** — word-token text replacement
3. **`meetingindex.md`** — row-scoped rewrite of all fields for the matching source
4. **Wiki** — `wiki/people/<old>.md` rename or merge into existing destination,
   plus cross-reference rewrite across all category pages, plus master index
   regen and in-memory search index rebuild

Plus an inline UI: click any speaker badge in a transcript, type a new name,
press Enter. If the speaker has multiple lines, a modal asks whether to fix
just the clicked line or rename all matches.

## The five iterations and what each one taught

### 1. Bulk only, transcript only
First version. `renameSpeakerInTranscript(content, from, to)` rewrote every
`[<from>]:` line in the `## Transcript` section. The `POST /note/speaker`
endpoint exposed it. Frontend bulk-updated the badges in place.

**What broke:** the wiki was completely out of sync. Renaming "Speaker 2" to
"Sean Song" in a transcript left `wiki/people/speaker-2.md` and dozens of
inline `Speaker 2` references in other wiki pages totally untouched.

### 2. Layered bulk rename (note + index + wiki)
Refactored into a coordinator with three pure helpers + an async wiki walker:

- `renameSpeakerInNoteContent` — rewrites both Transcript AND Summary sections
- `renameSpeakerInIndexContent` — rewrites the matching row's fields
- `renameSpeakerInWikiDir` — file rename or merge + walk-and-replace in
  every category, with `regenerateIndex` and `buildSearchIndex` afterwards

The merge case (destination people page already exists) appends the source
content under a `## Merged from "<from>"` audit marker and deletes the source.

**What this taught:** speaker rename is a data migration, not a single edit.
The right shape is a coordinator with per-layer counts in the response so the
caller can verify exactly what was touched.

**Two bugs caught by tests during this pass:**
- Step 2 of the wiki walker was rewriting its own merge marker because the
  marker contains the from-name verbatim. Fixed by tracking source AND
  destination in `skipPeopleFiles` and skipping both in step 2.
- Summary section regex used `(?=\n## [A-Z])` as the end marker, which
  wrongly terminated at nested LLM H2s like `## About Meeting`. Fixed to
  `(?=\n## Transcript\b)` matching `wikiCompiler.extractSummarySection`.

### 3. Dual-mode via collision heuristic
The user reported: "if one sentence was renamed, handle that, it should not
be the previous person's name and format". Single-line fix for misdiarized
sentences was missing.

Added a second mode:
- New name is **novel** (not in transcript) → BULK rename
- New name **collides** (already a speaker) → SINGLE-line fix

Backend got an optional `lineStart` parameter on `renameSpeakerInNoteContent`
to identify a single transcript line by `@<sec>`. Endpoint conditionally
skips summary/index/wiki when in single mode.

Frontend got a hint text below the input that updated live as the user typed.

**What broke:** the heuristic was wrong for the case where the SOURCE label is
a recognized name. Renaming a single Lex Fridman line to "Speaker 0" worked
correctly, but renaming a Lex Fridman line to "Ads Girl" was treated as bulk
(because Ads Girl is novel) and would rebrand all of Lex's lines.

### 4. Old-name pattern heuristic + Shift+Enter override
Switched the discriminator from new-name novelty to OLD-name pattern:

- `oldName` matches `^Speaker \d+$` (provisional cluster) → default BULK
- otherwise (recognized name) → default SINGLE
- Shift+Enter inverts the default

Better, but still required users to discover and remember the override.
Real-world test case from the user: a Lex Fridman + Jensen Huang interview
where the diarizer clustered ad audio into Lex's identity. Click any ad line
labeled `Lex Fridman`, type `Ads Girl`, the heuristic correctly picked SINGLE
mode and only the clicked line was rewritten with a fresh palette color.

**What this taught:** even a corrected heuristic still requires users to
either learn the rules or memorize the override gesture. The cognitive load
is wrong for a destructive operation.

### 5. Modal popup — explicit user choice
Final design. Removed the heuristic and the Shift+Enter override entirely.
Now:

- `matchCount === 1` → silently commit bulk (no choice to make)
- `matchCount > 1` → modal popup with three buttons:
  `Cancel · Just this line · All N lines`

The modal mirrors the existing delete-confirm overlay pattern (centered,
backdrop blur, scale animation). Esc cancels at any phase. Focus auto-moves
to the `All N lines` button so keyboard users can press Enter for the bulk
option.

The frontend uses a phase machine (editing → confirming → done) so blur,
keydown, and modal button clicks don't trample each other.

**Why this is the right answer:** the user knows their intent better than
any heuristic. Asking is cheap, getting it wrong is expensive.

## Tests

`tests/speakerRename.test.ts` — 35 tests covering:

- `buildSpeakerRegex` — whole-token, no partial-prefix, regex escaping, unicode
- `renameSpeakerInNoteContent` — rewrites both sections, preserves @timestamps,
  doesn't touch frontmatter, no-op on missing input, idempotent, handles
  labels without timestamp, handles missing transcript section
- SINGLE-line mode: only the matching @lineStart line, doesn't touch summary,
  no match returns 0, doesn't match other speakers on same lineStart,
  regex-escapes lineStart, idempotent, preserves @lineStart
- `renameSpeakerInIndexContent` — rewrites all fields on matching row only,
  unrelated rows untouched, idempotent
- `renameSpeakerInWikiDir` (uses real tmp directories) — file rename, merge
  with destination, walks all categories, partial-prefix protection,
  idempotent (twice == once), missing people page, unicode names, no-op when
  from===to

Total suite: **307/307 passing**.

## Lessons captured

Eight new entries in `tasks/lessons.md` under the 2026-04-08 heading. Key
ones:

- **Heuristics for user intent are unreliable** for destructive/propagating
  actions. Ask explicitly when there's ambiguity.
- **Single-instance and bulk modes for the same operation should have
  different propagation scopes**, not just a different match count.
- **Word-boundary regex via `\b` is unsafe when the from-string ends in a
  digit**. Use explicit non-word-class lookahead/lookbehind.
- **Backticks inside JS comments are LIVE inside a TypeScript template
  literal.** Use straight prose, not markdown code-quotes.
- **Test assertions for layered features must include the audit markers
  themselves**, not just the substantive content.
